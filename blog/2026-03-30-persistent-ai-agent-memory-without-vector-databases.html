We Gave Our AI Agent Persistent Memory Using SQLite. No Vector Database Required.

March 30, 2026 · Ceradon Systems

We run autonomous AI agents 24/7 for internal operations—business development, code review, email triage, competitive intelligence. After months of iteration, the single biggest reliability improvement wasn’t a better model or a fancier framework. It was giving the agent persistent memory that actually works. Here’s how we built it and why we deliberately ignored the conventional wisdom.

The Problem: Every Session Starts from Zero

Large language models have no persistent state. Every time your agent starts a new session, it wakes up with amnesia. It doesn’t remember that the client prefers email over phone. It doesn’t know that the last three attempts to authenticate with a particular API failed because of a token format issue. It doesn’t recall that you already tried approach A and it didn’t work.

The standard solutions are context windows (expensive, limited) and RAG pipelines with vector databases (complex, brittle). We tried both. Context stuffing burns tokens fast when your agent runs dozens of sessions per day. Vector databases introduced a dependency chain—embedding APIs, similarity thresholds, chunk sizing—that created more problems than it solved.

We needed something that a CLI-driven agent could call in one line, that didn’t require an API key or external service, and that would survive reboots, crashes, and deployment changes. The answer was embarrassingly simple.

The Architecture: SQLite + FTS5

The entire memory system is a single SQLite database with an FTS5 full-text search index. That’s it. No vector embeddings. No cosine similarity. No external services.

The schema stores five things per memory:

  • Content—the actual text to remember
  • Category—a fixed taxonomy (project, business, infrastructure, personal, etc.)
  • Tags—freeform comma-separated labels for cross-cutting concerns
  • Source—where the memory came from (which agent, which session)
  • Upsert key—a unique identifier for facts that change over time

The upsert key is the most important design decision. When you store a fact like “current project status: Phase 2” with the key vantage.status, any future store with the same key replaces the old value instead of creating a duplicate. This means the database stays clean without manual maintenance. Facts evolve; they don’t accumulate.

The CLI interface is deliberately minimal:

# Store a memory
python memory.py store "Client prefers email over phone" -c business -t "client,prefs" -k "client.contact_pref"

# Search memories
python memory.py search "client preferences" -c business -l 5

# Check database health
python memory.py stats

An agent doesn’t need an SDK, a library import, or an API call. It runs a shell command and gets results. This matters more than it sounds—every dependency you add to an agent’s tool chain is a dependency that can break at 2 AM when no one is watching.

Why FTS5 Over Vector Search

The AI community has largely converged on vector embeddings as the “right” way to do memory retrieval. Embed the query, embed the stored memories, find the nearest neighbors by cosine similarity. It’s elegant in theory. In practice, for agent memory specifically, it has problems:

  • Embedding API dependency. If your embedding service is down, your agent has no memory. We run agents overnight and on weekends. External API outages at 3 AM are not theoretical.
  • Semantic drift. Vector similarity retrieves things that are “about” the same topic, but agent memory often needs exact recall. “What was the API key format issue?” is better served by keyword search than by semantic proximity.
  • Cost at scale. When your agent stores 50–100 memories per day and searches before every major action, embedding costs add up. FTS5 is free, forever.
  • Debuggability. You can open a SQLite database and read it. You can grep it. You can write SQL against it. Try debugging why a vector search returned an irrelevant result when all you have is a 1536-dimensional float array.

FTS5 handles the 90% case—finding memories that contain the words you’re looking for, with ranking by relevance. For the remaining 10% where semantic search would genuinely help, the category and tag system narrows the search space enough that exact-match retrieval works fine.

The Workflow That Makes It Work

The database is necessary but not sufficient. What makes the system reliable is the discipline built into the agent’s operating instructions:

  • Search before storing. Every time the agent is about to store a memory, it first searches for duplicates. This prevents the database from filling with redundant entries.
  • Store actively, not passively. The agent doesn’t wait to be told to remember something. When it learns a new fact, encounters an error, or discovers a workaround, it stores it immediately. “Mental notes” don’t survive session restarts. SQLite does.
  • Search before answering. Before responding to questions about past events, the agent queries the database. This is the behavioral rule that provides the most value—it turns a stateless LLM into something that genuinely remembers.
  • Nightly consolidation. During active sessions, agents write quick notes to daily markdown files (fast, low-friction). A nightly cron job reads these files, extracts important facts, and stores them as structured memories in SQLite. This two-tier approach gives agents speed during the day and durability overnight.
  • Upsert for evolving facts. Project statuses, GPA, deployment dates, contact preferences—these change. Using upsert keys means the agent always retrieves the current value, not a stale one buried under newer entries.

What We Learned After 3 Months

The system has been running in production since January 2026, supporting 20+ autonomous agents across business development, software engineering, email management, and competitive intelligence. Some observations:

Categories matter more than tags. We started with a freeform tagging system. It became inconsistent within a week. Switching to a fixed category taxonomy (8 categories: military, project, business, infrastructure, coursework, legal, personal, financial) with freeform tags as a secondary index solved the consistency problem. Agents are surprisingly good at choosing from a fixed list. They’re terrible at inventing consistent taxonomies on their own.

The upsert key is the killer feature. Without it, the database would be 10x its current size and full of contradictory information. With it, every fact has a canonical location. When the agent searches for “current project status,” it gets one answer, not twelve historical snapshots.

500 memories is the sweet spot. Our database has a few hundred structured memories, not thousands. Quality and recency matter more than volume. The nightly consolidation cron is as much about pruning as it is about persisting.

Behavioral rules do more than clever retrieval. The biggest improvements came from adding two lines to the agent’s system prompt: “Search memory before answering questions about past events” and “Store important facts immediately, don’t make mental notes.” No architectural change had as much impact as these two behavioral instructions.

When This Approach Falls Short

To be clear about the limitations: this system works well for structured factual recall—what happened, what was decided, what failed, what someone prefers. It does not work as well for:

  • Semantic retrieval across large document corpuses (use a proper RAG pipeline for that)
  • Conversation history replay (we store summaries, not transcripts)
  • Multi-agent shared state with consistency guarantees (SQLite handles concurrent reads fine, but concurrent writes from multiple agents need WAL mode and careful design)

For our use case—a primary agent with multiple specialized sub-agents, all running on the same machine—SQLite with FTS5 handles everything we need. If we were running distributed agents across multiple servers, we’d need something more. We’re not, so we don’t.

The Point

The AI agent ecosystem is drowning in complexity. Every week brings a new memory framework, a new vector database, a new retrieval strategy. Most of them solve problems you don’t have yet while introducing problems you didn’t have before.

If you’re building agents that need to remember things across sessions, start with the simplest thing that could work: a SQLite database, full-text search, a CLI your agent can call, and a set of behavioral rules that tell the agent when to store and when to search.

You can always add vector search later if you need it. In three months of production use, we haven’t.
